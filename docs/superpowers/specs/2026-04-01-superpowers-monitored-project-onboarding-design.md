# Superpowers Monitored Project Onboarding Design

## Goal

Let the dashboard treat `useSuperpowers=true` as a control-plane declaration for monitored repos.

When this flag is enabled from the dashboard:

- attach, recovery, and workbench prompts must switch into Superpowers mode
- repo-local control assets injected by the dashboard must reflect Superpowers workflow constraints
- removing the monitored project from the dashboard must clean up dashboard-managed control assets without touching business code or user-owned repo content

This design does not change the previously approved aggregation truth order:

1. `/.codex-control/project_state.json` and `runs/*.json`
2. `docs/superpowers/specs/*.md` and `docs/superpowers/plans/*.md`
3. git / repo-visible inferred fallback evidence

## Problem Statement

The repo already has partial Superpowers support in prompt text and aggregation, but onboarding is still too loose.

Current gaps:

- `useSuperpowers` is mostly treated as a prompt hint instead of a repo onboarding mode
- attach / recovery does not clearly expose a stable Superpowers onboarding state
- repo-local injected assets do not consistently distinguish normal monitored repos from Superpowers-monitored repos
- cleanup on project removal should remove dashboard-managed Superpowers control assets, not leave stale monitored-project protocol files behind
- cleanup must still preserve user-owned files and business code

## Scope

### In Scope

- make `useSuperpowers=true` a stable monitored-project onboarding mode
- propagate that mode through attach, recovery sessions, and workbench prompt bundles
- strengthen dashboard-managed `AGENTS.md` block and repo-local skills for Superpowers repos
- optionally scaffold dashboard-owned `docs/superpowers/` entry files when missing
- record enough ownership metadata so cleanup removes only dashboard-managed files
- extend project removal cleanup to cover dashboard-managed Superpowers onboarding assets

### Out Of Scope

- changing monitored project business code
- forcing the dashboard to auto-detect Superpowers mode from repo contents
- changing steady-state aggregation truth priority
- adding new visual product flows beyond what existing workbench and onboarding already expose
- implementing automatic spec / plan generation for monitored repos

## Design Principles

1. Dashboard declaration is the source of truth for onboarding mode.
2. Repo facts still override chat descriptions inside steady-state aggregation.
3. Cleanup removes only dashboard-managed assets, never user-owned documents or business code.
4. Superpowers mode should be visible in prompt bundles, injected repo rules, and cleanup behavior, not hidden in one boolean only.

## User Model

There are two onboarding paths:

### New Attach

The user adds a monitored repo from the dashboard and enables `useSuperpowers`.

Expected result:

- the project is stored as a Superpowers-monitored repo
- dashboard-managed control assets are injected in Superpowers mode
- future Codex prompts generated by the dashboard explicitly require Superpowers workflow compliance when specs / plans exist

### Existing Project Recovery

The user attaches a repo for recovery and enables `useSuperpowers`.

Expected result:

- recovery session stores that mode
- recovery prompts instruct Codex to rebuild control files without bypassing Superpowers workflow expectations
- if the dashboard created missing `docs/superpowers/` entry scaffolding, that scaffolding is tracked as dashboard-owned

## Data Model Changes

### Project Registry Record

The monitored project record should keep:

- `useSuperpowers: boolean`
- `onboardingMode: "standard" | "superpowers"`

`onboardingMode` is derived from the dashboard choice and used anywhere attach / refresh / removal needs a stable mode value.

### Project Config / Ownership Metadata

Dashboard-managed repo config should record whether Superpowers onboarding assets were created by the dashboard.

Minimum tracked ownership data:

- whether dashboard created `docs/superpowers/README.md`
- whether dashboard created placeholder files under `docs/superpowers/specs/`
- whether dashboard created placeholder files under `docs/superpowers/plans/`

This metadata can live in existing repo-side config controlled by the dashboard.

The purpose is to support safe cleanup on project removal.

## Onboarding Behavior

### Attach / Add Project

When `useSuperpowers=true`:

- the project record is persisted with `onboardingMode: "superpowers"`
- scaffold generation uses Superpowers-aware assets
- API result should expose that the project is under Superpowers onboarding mode

Superpowers-aware scaffold means:

- stronger dashboard-managed `AGENTS.md` block
- Superpowers-aware repo-local skills
- optional minimal `docs/superpowers/` entry scaffolding when missing

### Recovery Attach

When `useSuperpowers=true`:

- recovery session persists `useSuperpowers` and `onboardingMode`
- prompt bundle must explicitly treat the repo as Superpowers-controlled
- recovery scan remains a control-state rebuild flow, not a business-implementation shortcut

If `docs/superpowers/` is absent and dashboard creates a minimal entry scaffold, that creation must be marked as dashboard-owned.

## Prompt Bundle Changes

Prompt bundles for Superpowers-monitored repos must move from a weak hint to a mode-aware instruction set.

### New Project Prompt Bundle

The generated Codex structure prompt must say:

- this repo is in Superpowers mode
- future boundary-changing work should follow `docs/superpowers/specs/** -> docs/superpowers/plans/** -> implementation`
- closeout must keep `project_state.json`, `runs/*.json`, and confirmed Superpowers decisions aligned

### Recovery Prompt Bundle

The recovery prompt must say:

- this repo is in Superpowers mode
- rebuild control files from repo facts first
- do not treat recovery guesses as approved design decisions
- if later implementation continues, workflow-defining changes should go through specs / plans

### Handoff / Closeout / Recovery Repo-Local Skills

Repo-local skills injected by the dashboard must reflect Superpowers mode:

- `codex-project-handoff`
- `codex-task-closeout-writeback`
- `codex-project-recovery-scan`

For Superpowers repos these skills must:

- mention `docs/superpowers/specs/**` and `docs/superpowers/plans/**` as workflow constraints when present
- avoid bypassing approved specs / plans through direct implementation
- keep source-state files aligned with confirmed Superpowers decisions

## Repo Asset Injection Rules

### Always Dashboard-Managed

These assets are dashboard-managed and should be cleaned on project removal:

- `/.codex-control/**`
- dashboard-managed block inside `AGENTS.md`
- dashboard-managed block inside `.gitignore`
- dashboard-injected repo-local skills under `.agents/skills/`

### Conditionally Dashboard-Managed

These assets should only be cleaned if the dashboard created them:

- `docs/superpowers/README.md`
- `docs/superpowers/specs/.gitkeep` or equivalent placeholder
- `docs/superpowers/plans/.gitkeep` or equivalent placeholder

If user-owned content already exists in those locations, the dashboard must not claim ownership of it and must not delete it on cleanup.

### Never Dashboard-Managed

These must never be deleted by removal cleanup:

- business code
- user-owned docs
- user-maintained `docs/superpowers/specs/*.md`
- user-maintained `docs/superpowers/plans/*.md`
- any non-dashboard `AGENTS.md` content

## Removal Behavior

When a monitored project is removed from the dashboard:

- remove `/.codex-control/`
- remove dashboard-managed repo-local skills
- remove dashboard-managed `AGENTS.md` block
- remove dashboard-managed `.gitignore` block
- remove dashboard-owned Superpowers entry scaffolding only if metadata says the dashboard created it

If `docs/superpowers/` contains user-owned content, preserve the directory and those files.

Removal must stay idempotent:

- safe if some files were already manually removed
- safe if user modified non-dashboard sections of `AGENTS.md` or `.gitignore`
- safe if only part of the dashboard-managed Superpowers scaffold still exists

## Implementation Shape

### Backend

- extend attach / recovery flows to persist stable onboarding mode
- extend scaffold generation to choose standard vs Superpowers-managed assets
- extend repo-local skill template generation for Superpowers mode
- extend removal cleanup to delete only dashboard-owned Superpowers entry scaffolding
- expose onboarding mode in attach / workbench responses where useful

### Documentation

- update mechanism map with the onboarding-mode model
- document which Superpowers repo assets are dashboard-managed versus user-owned

## Testing

### Required Cases

1. attach with `useSuperpowers=true` injects Superpowers-aware rules, skills, and tracked ownership
2. recovery attach with `useSuperpowers=true` returns Superpowers-aware prompt bundles
3. removing a Superpowers-monitored project deletes dashboard-managed control assets
4. removal does not delete user-owned Superpowers spec / plan files
5. standard monitored projects still behave as before

### Regression Focus

- existing standard attach flow must remain valid
- existing cleanup behavior for non-Superpowers projects must remain valid
- Superpowers mode must not cause steady-state refresh to write business or derived cache into monitored repos

## Risks

### Risk: Over-deleting user docs on removal

Mitigation:

- only delete files explicitly marked as dashboard-owned
- never recursively delete `docs/superpowers/` without ownership checks

### Risk: Prompt mode and injected repo rules diverge

Mitigation:

- centralize Superpowers-mode prompt fragments and skill/rules templates
- test attach and recovery prompt bundles directly

### Risk: `useSuperpowers` becomes a weak label again

Mitigation:

- expose stable onboarding mode in project records and attach results
- route scaffold, prompt, and cleanup behavior from the same mode source

## Acceptance Criteria

This feature is complete when:

- checking `useSuperpowers` in the dashboard reliably puts a monitored repo into Superpowers onboarding mode
- generated prompts, injected repo rules, and repo-local skills all reflect that mode
- removing the monitored project cleans dashboard-managed Superpowers assets
- user-owned repo files remain untouched
- standard monitored repos are unaffected
